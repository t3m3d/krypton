## 2. `heuristics.yml` Entry

The `.k` extension is shared with **KCL** and **KFramework**. Add a Krypton
rule **before** the existing KCL/KFramework rules so it is evaluated first.

Replace the current `.k` block:

```yaml
- extensions: ['.k']
  rules:
  - language: KCL
    pattern:
      - '^schema [A-Za-z0-9_-]+'
      - '^\}$'
      - '\s*\}\s*'
  - language: KFramework
    pattern:
    - '^requires\s+"[^"]+"$'
    - '^syntax\s+\w+\s+::=.*'
    - '^endmodule$'
```

With:

```yaml
- extensions: ['.k']
  rules:
  - language: Krypton
    pattern:
    - '^\s*just\s+run\s*\{'
    - '^\s*go\s+\w+\s*\{'
    - '\bemit\s+'
  - language: KCL
    pattern:
      - '^schema [A-Za-z0-9_-]+'
      - '^\}$'
      - '\s*\}\s*'
  - language: KFramework
    pattern:
    - '^requires\s+"[^"]+"$'
    - '^syntax\s+\w+\s+::=.*'
    - '^endmodule$'
```

### Pattern rationale

| Pattern | Description | Unique to Krypton? |
|---|---|---|
| `^\s*just\s+run\s*\{` | Entry point `just run { ... }` | Yes — no other `.k` language uses this |
| `^\s*go\s+\w+\s*\{` | Named entry point `go main { ... }` | Yes — Krypton-specific keyword |
| `\bemit\s+` | Return-like keyword `emit value` | Yes — not used in KCL or KFramework |

These patterns will not false-positive on KCL (which uses `schema`, `mixin`,
`protocol`, `rule`, `check`) or KFramework (which uses `requires`, `syntax`,
`endmodule`).

## 5. PR Description

Use the following when creating the pull request:

---

### Title

**Add Krypton programming language**

### Body

**Language name:** Krypton
**Extension:** `.k`
**Type:** programming
**Color:** `#7B42BC`
**Grammar scope:** `source.krypton`

#### What is Krypton?

Krypton is a self-hosting programming language with a C-like syntax designed
for systems programming and compiler bootstrapping. It features:

- First-class functions (`func name(args) { ... }`)
- Entry points (`just run { ... }`, `go main { ... }`)
- `emit` as the return mechanism
- `let` bindings, `while` loops, `if`/`else` control flow
- String operations, arithmetic, and I/O builtins
- A self-hosting compiler that compiles Krypton to C

The language is fully self-hosting: the Krypton compiler is written in Krypton
and compiles itself to C, which is then built with a native C compiler. The
compiler has achieved a fixed-point (the compiler compiles itself to produce
byte-identical output).

#### Repository

https://github.com/t3m3d/krypton (MIT license)

#### Grammar

TextMate grammar (`source.krypton`) is at:
[`krypton-lang/syntaxes/krypton.tmLanguage.json`](https://github.com/t3m3d/krypton/blob/main/krypton-lang/syntaxes/krypton.tmLanguage.json)

#### Extension disambiguation

`.k` is shared with KCL and KFramework. Heuristic patterns added:
- `^\s*just\s+run\s*\{` — Krypton entry point (unique)
- `^\s*go\s+\w+\s*\{` — Krypton named entry point (unique)
- `\bemit\s+` — Krypton return keyword (unique)

These do not conflict with KCL (`schema`, `mixin`, `check`) or
KFramework (`requires`, `syntax`, `endmodule`).

#### Samples

Five representative `.k` files included in `samples/Krypton/`, all derived
from real compiler source code (tokenizer, parser, evaluator, features demo,
source analysis tool).

## 6. Known Considerations

### Adoption threshold

Linguist requires **200+ unique `.k` Krypton files on GitHub** (or **2,000+**
since `.k` is a commonly used extension). The current repo has ~41 `.k` files.

**Strategies to meet threshold:**
1. Split examples into more individual files
2. Encourage community adoption and more Krypton repos
3. Publish the standard library as separate `.k` files
4. Create tutorial/example repos with `.k` files

The PR may be rejected initially on adoption grounds. Consider growing the
file count before submitting, or submit early to get feedback on the technical
aspects (grammar, heuristic, samples).

### Grammar repository

Linguist prefers grammar repos that can be vendored. Options:
1. **Use main repo** — grammar at `krypton-lang/syntaxes/krypton.tmLanguage.json`
2. **Create dedicated repo** — e.g., `t3m3d/krypton-tmLanguage` with just the
   grammar file. This is cleaner for linguist's `script/add-grammar` workflow.
