### Body

```
Krypton is a self-hosting language I’ve been building solo. The
compiler is written in Krypton and compiles itself. No LLVM, no GCC at
user-invocation time. Three hand-rolled native backends emit ELF on
Linux, PE on Windows, and Mach-O on macOS arm64 directly. The macOS
backend writes its own ad-hoc SHA-256 code signature so the binary
runs under Tahoe AMFI without going through clang/ld/codesign.

This week the fourth target landed: WebAssembly. The proof-of-concept
emits a valid 95-byte hello.wasm from ~140 lines of Krypton, loads via
WebAssembly.instantiate, and prints "Hello". Phase 0 was just confirming
the design works on Krypton’s current runtime; phase 1 (IR → WASM
lowering) is the next ~2 weeks.

What the proof looks like (hex dump of the emitted module):

  00 61 73 6d 01 00 00 00 01 09 02 60 02 7f 7f 00
  60 00 00 02 13 01 03 65 6e 76 0b 63 6f 6e 73 6f
  6c 65 5f 6c 6f 67 00 00 03 02 01 01 05 03 01 00
  01 07 13 02 06 6d 65 6d 6f 72 79 02 00 06 5f 73
  74 61 72 74 00 01 0a 0a 01 08 00 41 00 41 05 10
  00 0b 0b 0b 01 00 41 00 0b 05 48 65 6c 6c 6f

That + Krypton's existing backends gets you four targets on three OSes
and the browser, all from a self-hosting compiler that fits in roughly
5K lines.

The language has a one-EBNF-page spec, familiar surface (func/fn,
if/elif/else, while, for-in, match, try/catch, lambdas, structs), and
~150 builtins. The whole web stack is in Krypton too:

- htmk — HTML emit DSL
- kcss — CSS emit DSL (new this week)
- ks — KryptonScript → JS DSL
- k:server / k:router — Express-style HTTP server in pure Krypton

krypton-lang.org is itself built with the framework. Every page is
generated by one export.htk. The 31 lesson pages each have an in-browser
"▶ Run" button (currently a JS bridge that handles 27/31 lessons; the
WASM backend retires it). You can poke around at
https://krypton-lang.org/playground.html .

Code: https://github.com/t3m3d/krypton
Site / lessons / playground: https://krypton-lang.org
WASM design + phase 0 finding: https://github.com/t3m3d/krypton/blob/main/docs/wasm_backend_design.md
ELF backend free chapter: https://krypton-lang.org/blog/2026-06-01-elf-from-scratch.html

Happy to answer questions about the Mach-O signer, the self-hosting
bootstrap, the WASM emitter, or where the framework is headed next.
```

## Why the WASM hook works on HN

The audience that reads PL posts on HN has seen a hundred "Show HN: my
small language" posts. Most never get past 30 upvotes. The difference
between "small lang" and "interesting small lang" is usually one
genuine differentiator that takes work to do.

For Krypton, that differentiator is **self-hosted with no LLVM**. Most
Show HN languages either (a) AST-walk an interpreter, (b) emit C and
call gcc, or (c) sit on top of LLVM. Krypton emits the actual binary
formats from hand-written backends. That's rare enough to be
noteworthy.

The WASM proof makes it concrete in a way the existing backends
don't: WASM is the format HN readers care most about (browser, edge,
serverless), and the proof binary is small enough (95 bytes!) that the
hex dump fits in the post body. That's the visual hook that converts
scroll-past into click.

Make sure the title says "WebAssembly" or "WASM" explicitly — that's
the keyword that earns the click. "Native code" is too vague.
